# v0.6 plan — cgrca as the structural layer

> Status: design draft. Builds on the v0.5 honest re-positioning (see
> `CHANGELOG.md` v0.5.0 entry and `docs/v0.5-plan.md` Phase 4 verdict).

## Why this plan exists

The v0.5 plan tested whether "graph-walk RCA + LLM re-rank > embedding
retrieval + LLM" on a labelled corpus. It doesn't (at least not on the
8-bug the eval corpus corpus 2026-05-03). cgrca's `--llm` Sonnet hit top-1 = 0.750
vs the `@codebase`-style baseline at 0.875 — a -12.5pp gap where the v0.5
plan's pre-commit said cgrca needed to win by +10pp.

The v0.5 plan said "pivot or stop." This plan is the pivot.

## The honest positioning

cgrca's actual moat isn't retrieval. Embedding tools (Cursor `@codebase`,
GitHub Copilot, Continue, …) will always have:

- Better embeddings (they trained them on billions of tokens)
- Bigger scale (no file budget; no per-symbol indexing)
- Adaptive retrieval (the LLM picks what to read)

cgrca has things they fundamentally can't have at this granularity:

- Call graph: callersOf, calleesOf, pathBetween, transitive reach
- Git signal: recentlyChangedNear, co-change history per symbol, blame
- Calibrated scoring: recency × proximity × ambiguity × co-change ×
  subsystem × complexity × dataflow, fit by logistic regression against
  a labelled corpus
- Per-repo persistence + fs-watch invalidation (cgrcad daemon)
- Symbol-level granularity with kind / loc / subsystem metadata

These are **ground truth from AST + git**, not approximate from
embeddings. Together they form the *structural layer* for code-RCA — the
layer that turns "this file looks similar to your prose" into "this
function changed 2 days ago, is called from 47 places, co-changes with
the failing test in 12 of the last 30 commits."

**v0.6 ships the tools that make this composable with any retriever.**

## Phase 6 — `cgrca_enrichCandidates` MCP tool

**Effort**: 3-5 days.
**Risk**: Low — composes existing primitives.

### What it does

Accepts a list of `(file, symbol)` candidates from ANY retriever (Cursor
`@codebase`, embedding RAG, BM25, even hand-typed) and returns each
candidate annotated with cgrca's structural signals.

### Surface

```ts
// MCP tool input schema
{
  candidates: Array<{ file: string; symbol?: string; line?: number }>,
  failure?: string,    // optional — used to compute proximity / score
  topK?: number,       // re-rank and return top-K (default: keep input order)
  includeBody?: boolean,    // default true; first 30 lines per symbol
  includeNeighbors?: boolean, // default true; 1-hop callers/callees
  includeRecency?: boolean,   // default true; last 5 commits per symbol
}
```

```ts
// Output: each candidate enriched
{
  enriched: Array<{
    file: string;
    symbol: string | null;       // resolved if input symbol/line was partial
    line: number | null;
    kind: SymbolKind | null;
    subsystem: string | null;
    loc: number | null;          // body length in lines

    // Structural signals (the actual product)
    signals: {
      recencyScore: number;           // 0..1, days since last commit decayed
      proximityScore: number;         // graph distance to failure scope
      ambiguityScore: number;         // unique-name / co-name penalty
      coChangeScore: number;          // % commits also touching failure scope
      subsystemScore: number;         // same package as failure
      complexityScore: number;        // body length / branching
      dataflowScore: number;          // pathBetween over arg-flow
    };
    compositeScore: number;           // calibrated linear combo (0..1)

    // Context the LLM would otherwise have to ask for
    body: string | null;              // first 30 lines, optional
    callers: Array<{ file: string; symbol: string; line: number }>; // 1-hop
    callees: Array<{ file: string; symbol: string; line: number }>; // 1-hop
    recentChanges: Array<{ sha: string; date: string; subject: string; author: string }>;

    // Provenance — explain why this candidate ranks here
    rationale: string;
  }>;
}
```

### Why this works for v0.6

- **Compositional with any retriever.** Cursor user runs `@codebase`,
  pipes the result into `cgrca_enrichCandidates`, gets structural
  evidence the LLM uses to pick the right one. Same for Copilot,
  Continue, Aider, hand-grep, anything.
- **No competition with embedding retrievers.** They keep doing what
  they're good at; cgrca adds what they can't.
- **The agent's existing UX stays.** Cursor users keep typing
  `@codebase`. They just get better answers because the agent now also
  calls `cgrca_enrichCandidates`.
- **Reuses every primitive cgrca already has.** This is plumbing, not
  re-architecture: `definitionOf` + `callersOf` + `calleesOf` +
  `recentlyChangedNear` + `buildCausalChain` (the existing scorer) +
  `fetchBody` (from `--llm` work) all wired together.

### Eval gate

- Compose `llm-codebase` (BM25 retriever) → top-K candidates → run
  through `cgrca_enrichCandidates` → LLM picks from enriched set
- Compare against bare `llm-codebase` (no enrichment) on the same corpus
- **Pass criterion**: enrichment lifts top-1 by ≥5pp on the 8-bug
  the eval corpus corpus, AND ≥10pp on a 30-bug expanded corpus

If enrichment doesn't lift retriever-only by ≥5pp on either corpus, the
"structural signals add value when wrapped" bet is wrong. Stop and
re-think.

## Phase 7 — Self-improvement loop (was Phase 5 in v0.5 plan)

**Effort**: 1 week.
**Risk**: Low — but value depends on Phase 6 working.
**Conditional**: Only ship if Phase 6 eval shows real signal.

### Why this is the actual moat

Embedding retrievers are calibrated ONCE on a generic corpus during
training. They don't adapt to a specific repo's idioms, naming
conventions, common bug patterns, or signal weights. cgrca CAN — it has
the substrate (per-repo persistent SQLite, MCP tool call log, github-app
PR comment audit trail).

Per-repo calibration is something Cursor / Copilot / GitHub structurally
can't do without rebuilding their retrieval architecture around per-repo
state.

### What ships

- **Observation log**: every `cgrca_rcaWithReasoning` call (and every
  github-app PR-bot comment) writes a tuple to
  `~/.cgrca/repos/<sha>/observations.jsonl`:
  ```jsonl
  {"ts":"2026-05-10T14:23:00Z","failure":"...","candidates":[...],"merged_fix_files":["core/x.py"],"merged_fix_symbols":["foo"]}
  ```
- **`cgrca calibrate --from-observations`**: reads the log, computes
  per-signal hit-rate against the merged fixes, optionally re-fits the
  causal-scorer weights (logistic regression). Shows a "this run
  shifted weights by X% — apply?" gate so re-fits aren't silent.
- **`cgrca observations --misses`**: the cases the bot consistently
  fails on. This is the next-iteration backlog, generated empirically.
- **Per-repo few-shot library** (for `cgrca_rcaWithReasoning` LLM
  prompt): top-K past wins extracted as examples, attached to future
  prompts. Cursor can't replicate this — they don't keep per-repo
  observation logs.

### Eval gate

- After 4 weeks of bot operation on a real repo (the eval corpus or cgrca
  itself), compare the calibrated weights' top-1 vs the v0.5 default
  weights on a holdout.
- **Pass criterion**: ≥3pp top-1 lift after 4 weeks of observation.

If it doesn't lift, observations are noisy / corpus too small / signal
weights aren't load-bearing. Either invest in better label extraction
(LLM-grade each observation: "is this PR really a bug fix?") or accept
the loop as a "data archive" (still useful for postmortems) and stop
auto-recalibration.

## What we explicitly DON'T do in v0.6

- **No new retriever.** v0.5's free-text + matcher widening is enough as
  a zero-config fallback. Don't compete with embedding tools at their
  game.
- **No `--llm` flow improvements** beyond what v0.5 already ships.
  Sonnet 4.6 + the existing prompt + the v7 substrate is the floor;
  improving it further is research-y and high-risk.
- **No tagged release until both gates pass.** v0.6.0-rc.N until Phase
  6 eval shows signal AND Phase 7 calibration loop shows lift over the
  4-week soak window. Then v0.6.0 GA.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Phase 6 enrichment doesn't lift retriever-only top-1 | Medium | High | 5pp gate on the eval corpus + 10pp on 30-bug corpus. If it fails, the structural-signal bet is wrong; reposition cgrca as "the daemon + queries you can call from any agent" without product framing. |
| Phase 7 observation log too noisy | High | Medium | LLM-grade labels (`cgrca calibrate --grade-with-llm`); accept multi-month soak window |
| Cursor / Copilot adds graph-style signals natively | Medium | High | Their architecture isn't built for it (per-repo persistent state). They'd need to re-architect. We have a head start; ship faster. |
| 8-bug corpus too small to signal anything | High | Medium | Grow corpus to 30+ bugs across 3+ repos before final pass/fail. |

## Sequencing

```
Phase 6 (3-5d) — cgrca_enrichCandidates MCP tool
   ↓ (eval gate)
Phase 7 (1w) — self-improvement loop (per-repo calibration)
   ↓ (4-week soak gate)
v0.6.0 GA
```

Total: ~2-3 weeks of agent time + 4 weeks of soak window for Phase 7
calibration. ~7 weeks calendar end-to-end.

## What the v0.5.0 release commits to publicly

- `--llm` is positioned as a zero-config fallback, not the headline
- README + CHANGELOG say cgrca is the "structural layer" composed with
  any retriever, not a replacement for embedding retrieval
- v0.6 plan is published alongside v0.5.0 so users see the trajectory
- The eval numbers (cgrca 0.750 vs llm-codebase 0.875) are published
  honestly, with an explanation of why we ship anyway
