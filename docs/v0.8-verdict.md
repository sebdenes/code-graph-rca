# v0.8 — graph-traversal eval and the verdict on cgrca's RCA pipeline

## TL;DR

Across four eval phases on three different framings of the retrieval task,
cgrca's integrated RCA pipeline lost to simpler baselines every time:

| Phase | Eval | cgrca | Baseline | Verdict |
|---|---|---|---|---|
| v0.5 | 9-bug prose-to-file (cherry-picked) | 0.444 | grep 0.333 | parity |
| v0.6 | LLM-augmented enrichment | -25 to -33pp | embeddings | failed gate, scrapped |
| v0.7 | 29-issue prose-to-file (broader) | 0.103 | grep 0.207 | lost 2:1 |
| v0.8 | graph-traversal (cgrca's actual job) | 0.062 / 0.217 | cochange 0.595 / 0.498 | lost 2-10x |

(v0.7 number is text top-1; v0.8 numbers are top-10 / top-5 against `git log` cochange aggregation. v0.8 reports both pre-fix and post-fix cgrca to show the cochange-scope expansion lifted cgrca 3.5x but still didn't catch up.)

The pre-committed v0.8 kill criterion: ship if cgrca-with-cochange-scope hits ≥0.4 top-5 OR beats naive cochange. Neither was met.

## Method (v0.8 graph-traversal eval)

The first three eval phases all measured prose-to-file retrieval — which is grep's home turf, not cgrca's. v0.8 was built to test cgrca on its **designed job**: given the first file an engineer touches in a fix, can cgrca's graph reach the other files in the fix?

- Mined the source repo's git history: 593 commits touching 2-6 .py files, filtered to 295 where all files still exist at HEAD.
- Sampled 30-80 commits.
- Per commit: seed = first file, targets = remaining files. Scored top-1/top-5/top-10 coverage of targets in each method's ranked output.
- Methods: cgrca rca file:<seed>, random files (control), same-dir files (cheap heuristic), git-cochange (top files that historically co-changed with seed via `git log --name-only`).

Results (n=80 baseline, n=30 post-fix):

```
                v0.7 baseline                v0.8 cochange-scope
method        top-1   top-5  top-10        top-1   top-5  top-10
cgrca         0.050   0.062  0.062         0.083   0.217  0.217
random        0.000   0.022  0.022         0.000   0.053  0.059
samedir       0.013   0.038  0.110         0.000   0.044  0.150
cochange      0.292   0.565  0.659         0.126   0.498  0.595
```

## Root cause

cgrca's scope was built **structurally first** (imports + callers + callees of the seed file's symbols), then ranked within that scope using a 7-signal scorer in which coChange has the highest weight (0.47 of the calibrated weights). But coChange could only rerank what the structural walk had already admitted to the scope.

Most multi-file fixes in real repos involve files that are **not** structural neighbors:
- `config.py` and `scheduler.py` (config defines what scheduler reads)
- parallel templates (`prompts/morning_checkin.md` and `prompts/evening_checkin.md`)
- coordinated data-shape changes across `agent.py` and `targets.py`

These never entered the structural scope, so they could never be ranked, no matter what the scorer did.

The v0.8 architectural fix in this PR: cochange-expanded scope. Files that historically co-changed with the seed now enter the scope (and seed list) directly, before any structural traversal. This let cgrca's ranking finally surface them — lifting cgrca's top-5 from 0.062 → 0.217, a 3.5x improvement.

But naive `git log --name-only` aggregation still beats cgrca by 2.3x on top-5 and 2.7x on top-10. **The structural walk wraps lossy reduction around cochange data; the wrapping costs more than the structural signal adds.** Cochange seeds with no symbols (markdown, configs) are dropped. Cochange files whose symbols don't reach targets via callers/callees never enter `causalCandidates`. The ranking pipeline reliably loses information that the raw cochange tally retained.

## What this PR ships

Code changes:
- `packages/core/src/graph/scope.ts`: `findCoChangedFiles()` helper + cochange-based scope expansion in `resolveScope`.
- `packages/core/src/rca/runner.ts`: multi-anchor walk now seeds from ALL scope seeds (original + cochange-expanded), not just the first.
- `packages/core/src/config/scoring-config.ts`: 3 new retrieval knobs (`cochange_seed_topn`, `cochange_lookback_days`, `cochange_min_cooccurrences`).
- `tools/calibration/scoring-config.json`: defaults for the new knobs.

This is shipped because:
- It's a correctness improvement (cgrca's strongest signal was being gated). Even though the integrated pipeline still loses, the fix is the right thing to do for any code path that uses scope expansion.
- It documents a definitive eval result that wouldn't exist without it.
- 351/351 tests pass.

## What dies

- "cgrca rca <prose>" as a competitive prose-to-file retriever — it loses to grep.
- The 7-signal causal scorer's "structural walk → score → rank" pipeline as the headline product — it loses to `git log` aggregation.
- The v0.6 LLM-enrichment narrative — already killed in v0.6.
- The v0.7 autoresearch-on-scoring-config narrative — saturated at one tiny improvement (substring_name_min_length 4→3, +3.9pp on the 9-bug corpus only).

## What survives

- The MCP tools (`callersOf`, `calleesOf`, `recentlyChangedNear`, `definitionOf`) — useful for agents that already have a seed and want structural enrichment per-symbol.
- The cgrca daemon, indexer, and sqlite layer — solid infrastructure.
- The 29-issue corpus and the graph-traversal eval harness — reusable for any future retrieval work in this space.
- The autoresearch loop infrastructure (scoring-config single editable surface, run-one.sh, log/best.json) — generalizes beyond cgrca; the loop ran cleanly, the verdict is just that this surface has limited room.

## Recommendation

The honest read of four eval phases is: **the right product on this corpus is naive git cochange aggregation + LLM rerank**, not the integrated cgrca RCA pipeline. cgrca's value is the structural enrichment LAYER on candidates that a different retriever surfaced — not the retriever itself.

Future direction (not in this PR): reposition cgrca as the structural-signal MCP tool layer for agents using their own retrieval. Replace `cgrca rca <prose>` in the README/positioning with `cgrca` as the daemon + tool surface other tools call into. Document the eval evidence behind the repositioning so it's clear this is an earned decision, not a pivot for its own sake.
