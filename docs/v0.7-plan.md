# v0.7 plan — autoresearch loop for cgrca

> Status: design draft. Replaces v0.6 Phase 6 (enrichCandidates), which
> failed its eval gate. Inspired by [karpathy/autoresearch](https://github.com/karpathy/autoresearch).

## Why this plan exists

v0.6 Phase 6 hypothesised that **handcrafted structural enrichment** (call
graph + git history + body previews wrapped around BM25 retrieval) would
lift LLM RCA. Two iterations, both regressed top-1 by 33pp vs the bare
BM25 baseline. The structural facts confused Sonnet more than they helped.

Two honest readings of that result:

1. The handcraft was wrong — different enrichment shape might work.
2. **Handcrafting is the wrong vehicle.** The design space (which signals,
   what weights, what prompt structure, when to fire) is too large for
   manual exploration. We tried two of N possible designs; both failed; N
   is large.

This plan picks reading #2. Instead of handcrafting better heuristics,
build the autonomous loop that discovers them.

## The Karpathy autoresearch pattern

Per [karpathy/autoresearch](https://github.com/karpathy/autoresearch):

- **Single editable surface.** Agent only modifies one file (`train.py`
  in nanochat; one config / one weights file for cgrca).
- **Fixed time budget per experiment.** 5-minute training run in nanochat;
  one eval-corpus run in cgrca (~2-3 min).
- **One metric.** `val_bpb` in nanochat; `top-1` (or MRR) on a held-out
  corpus in cgrca.
- **Autonomous keep/discard loop.** Agent runs experiment → reads metric →
  keeps the change if metric improved → discards otherwise → proposes
  next change.
- **Human iterates on `program.md`**, not on the code being optimised.

The pattern is sharp because it forces the agent into a narrow,
measurable, comparable workspace where progress is visible and reversible.

## What we autoresearch in cgrca

The single editable surface: **`tools/calibration/scoring-config.yaml`**
(new file). One YAML, all the knobs the existing causal scorer + textmode
matcher use:

```yaml
# tools/calibration/scoring-config.yaml — the autoresearch surface
matcher:
  name_match_weight: 3.0
  subname_match_weight: 1.0
  signature_match_weight: 2.0
  body_content_weight: 0.3
  import_match_weight: 0.5
  body_content_min_length: 8
  substring_name_min_length: 4
  per_token_substring_cap: 100

scorer:
  recency_weight: 0.7
  proximity_weight: 1.2
  ambiguity_weight: 0.4
  cochange_weight: 0.9
  subsystem_weight: 0.3
  complexity_weight: 0.5
  dataflow_weight: 0.0  # currently zero per v0.4 calibration

augmenter:
  seed_cap: 8
  per_name_cap: 3
  matcher_tail_multiplier: 5

retrieval:
  default_max_files: 200
  default_top_n: 10
```

This config is read at `runRca` time (replacing the hardcoded constants
currently scattered across `causal.ts`, `textmode.ts`, `runner.ts`).

Why a YAML and not source edits: agent diffs are tiny + reviewable, and
the experiment is "what numbers in this file produce the best metric on
the eval corpus" — a search problem, not a programming problem.

## The autoresearch program

`tools/autoresearch/program.md` (new file) — the agent's instructions.
Modeled on Karpathy's `program.md`:

```
# program.md — cgrca autoresearch

You are running an autoresearch loop on cgrca's scoring config. Your goal
is to find the values in `tools/calibration/scoring-config.yaml` that
maximise `top-1` on the labeled eval corpus at `tools/eval/corpus-eval-fixed.jsonl`.

## How to run one experiment

1. Edit `tools/calibration/scoring-config.yaml` — change one or more knob
   values. Keep changes small (typically one knob per experiment unless
   you have a hypothesis why two should move together).
2. Run `npm -w packages/core run build` to rebuild cgrca with the new config.
3. Run `bash tools/eval/run-llm.sh` (with TARGET_REPO + ANTHROPIC_API_KEY
   set in env) — measures top-1 across all modes.
4. Read the printed metric. Compare against the previous best.
5. If top-1 improved by ≥1pp on the `text` or `llm` mode (the cgrca-
   contributed modes), commit the config change with a one-line message
   explaining the hypothesis and the delta. If not improved, revert
   `scoring-config.yaml` to the previous best and try a different knob.
6. Append a row to `tools/autoresearch/log.jsonl` with the experiment ID,
   knob changed, before/after value, before/after metric, kept/discarded.
7. Repeat.

## Constraints

- One experiment = one eval run (~2-3 min real, ~$0.30 LLM cost).
- Plan for ≤30 experiments per overnight run (~$10 budget cap).
- Touch ONLY `tools/calibration/scoring-config.yaml`, never the source
  code or the eval corpus.
- The eval corpus is held-out — never optimise against per-bug rank;
  only against the aggregate top-1 / MRR.
- If a single knob has been tried at 5+ values without improvement, mark
  it "saturated" in `tools/autoresearch/saturated.txt` and don't touch it
  again this run.

## What to optimise

Primary metric: `text` mode top-1 (the static cgrca scoring path).
Secondary: `llm` mode top-1 (catches cases where re-rank quality changes).
Tiebreaker: MRR.

Don't optimise against `llm-codebase` or `llm-codebase-enriched` — those
don't read scoring-config and won't move regardless.

## When to stop

- 30 experiments completed
- OR $10 LLM budget exhausted
- OR no improvement in 10 consecutive experiments

Print a summary table on exit. Commit all kept changes; discard the rest.
```

## Phase 7 — autoresearch infrastructure

**Effort**: 3-5 days
**Risk**: Medium — the loop has to actually find improvements, which
depends on the hypothesis space being well-shaped

### Ships

- **`tools/calibration/scoring-config.yaml`** — single editable surface.
  Default values match what's hardcoded today (so initial run = current
  behavior).
- **Refactor**: replace hardcoded scorer/matcher constants in `causal.ts`,
  `textmode.ts`, `runner.ts` with reads from the YAML at process start.
  No runtime overhead — read once, cached as constants.
- **`tools/autoresearch/program.md`** — agent instructions (above).
- **`tools/autoresearch/log.jsonl`** — append-only experiment log.
- **`tools/autoresearch/best.json`** — current best config + metric, for
  the agent to compare against.
- **`tools/autoresearch/run.sh`** — convenience wrapper:
  ```bash
  cd <repo> && time (npm -w packages/core run build && bash tools/eval/run-llm.sh)
  ```

### Eval gate

After 30 experiments on the 9-bug corpus:
- `text` mode top-1 must improve by ≥3pp from baseline (0.556 → ≥0.586)
- OR `llm` mode top-1 must improve by ≥3pp (0.778 → ≥0.808)
- AND no regression on the other modes

If neither metric improves by 3pp after 30 experiments, the autoresearch
loop isn't finding signal — either the knob space is wrong (try different
parameters), or the corpus is too small (grow corpus first), or the
scoring-config layer doesn't have enough degrees of freedom.

## Why this beats handcrafted enrichment

- **Wider search**: 30 experiments overnight vs 2 designs in a session
- **Empirical**: the loop discovers what works on YOUR corpus, not what
  the engineer thinks should work
- **Reversible**: every kept change has a measured improvement; every
  discard is logged
- **Compositional**: the human iterates on `program.md` (what knobs the
  agent can touch, what experiments are worth running) — much higher
  leverage than iterating on the source code being tuned

## What we explicitly DON'T do in v0.7

- **No new MCP tool work** until the autoresearch loop proves it can
  improve the static scorer. Tool design follows from "we know what
  signals matter" — autoresearch is what tells us that.
- **No corpus growth** until the autoresearch loop saturates on the
  current 9-bug corpus. If 9 bugs aren't enough to discriminate signal,
  more bugs will only stretch the run time.
- **No model upgrades** (Opus, etc.) — autoresearch should be model-cheap;
  Sonnet at $0.30/run is the budget unit.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Autoresearch finds local maxima only | Medium | Medium | program.md should encourage occasional "wild" perturbations (e.g. flip a knob 5×) to escape |
| 9-bug corpus too small to discriminate | Medium | High | Run autoresearch first; if no improvement after 30 experiments, grow corpus and re-run |
| LLM budget runs over | Low | Low | $10 hard cap in program.md; agent stops |
| Agent edits something other than scoring-config | Low | High | program.md says "ONLY edit scoring-config.yaml"; we trust the agent's discipline (Karpathy's program.md does the same) |
| Refactor introduces regressions | Medium | High | Default scoring-config matches current hardcoded values exactly. Re-run eval pre/post refactor — must hit identical numbers |

## Sequencing

```
Phase 7a (1 day) — extract scoring-config.yaml + refactor reads
Phase 7b (1 day) — write program.md + run.sh + log infrastructure
Phase 7c (overnight × N nights) — run autoresearch
   ↓ (eval gate: ≥3pp improvement)
v0.7.0 GA with the discovered config baked in
```

If the gate doesn't pass: document the experiments, ship the
infrastructure regardless (it's reusable), and re-think.
