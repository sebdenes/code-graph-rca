# v0.5 Phase 2 — LLM-augmented RCA (design)

> Status: design draft. Implementation gated on Phase 1 landing (PR #27) +
> the kill-criterion review in `docs/v0.5-plan.md`.

## Why this phase exists

Phase 1 (free-text retrieval) shipped 2026-05-02. On the 8-entry the eval corpus
corpus it hits **top-1 = 0.375** — same as naive grep, **worse on top-5**
(0.375 vs grep's 0.625). The retrieval surfaces relevant code, but it
doesn't pick THE right symbol at #1.

The Phase 1 audit established:
- The graph itself is solid (file mode hits 0.500 top-1)
- Retrieval is competitive at top-5 only when the question is "which
  symbol IN this file" (file mode), not "which file"
- Pure name-match retrieval can't disambiguate between 5 plausible
  candidates that all contain the same tokens

This phase tests whether **LLM re-ranking on top of cgrca's narrowed
candidate set** can lift top-1 enough to be worth the per-call cost. The
core bet: graph-narrowed top-10 (cheap) + LLM judgment on the narrowed
set (small token budget) > embedding retrieval on the whole repo.

## Kill criterion (the one that decides everything)

`cgrca rca --llm` must beat **"paste failure description + whole repo
to an LLM with `@codebase`-style retrieval"** by ≥10pp top-1 on the
same corpus.

If it doesn't: graph-walk retrieval doesn't add value over embedding
retrieval. Cursor wins this race. We pivot or stop — do NOT ship Phase 3.

This is in `docs/v0.5-plan.md` already. Re-stating here so the gate is
visible at implementation time.

## Surface area

### CLI

```
cgrca rca <failure> --llm [--provider anthropic|openai] [--model <name>]
                          [--max-llm-tokens 10000] [--no-llm-fallback]
```

- `--llm` opts in. Default off — Phase 2 is experimental until eval passes.
- `--provider` defaults to `anthropic`. `openai` covers OpenAI-compatible
  endpoints (Together, Groq, local llama.cpp servers).
- `--model` defaults per-provider (`claude-sonnet-4-6` for anthropic;
  `gpt-4o-mini` for openai).
- `--max-llm-tokens` caps the total context sent (input). Default 10000.
- `--no-llm-fallback` makes a network/auth failure exit non-zero rather
  than silently degrading to non-LLM mode.

The MCP-host path (Phase 3) won't need API keys — that's why CLI is
opt-in here, not auto-on. CLI users pay; MCP users use the host's LLM.

### Pipeline

```
input failure description
   ↓
Phase 1 free-text RCA → top-10 candidates (file, symbol, line, why)
   ↓
hydrate each candidate:
   - body preview (~30 lines from the source file, sliced
     [startLine..endLine] from definitionOf — NEW helper, not yet in
     `queries.ts`. Cap at 30 lines, mark truncation visible in prompt.)
   - 1-hop callers (top 3 by score; reuses `callersOf`)
   - 1-hop callees (top 3 by score; reuses `calleesOf`)
   - definitionOf for the symbol (already exists)
   ↓
build prompt:
   - failure description (verbatim)
   - candidate set (rendered as a structured block per candidate)
   - explicit instruction: pick the most likely root cause + why
   - explicit output schema (JSON)
   ↓
LLM call with --max-llm-tokens budget
   ↓
parse response → { rootCause: { file, line, symbol, hypothesis,
                                confidence }, alternatives: [...] }
   ↓
return: LLM verdict + the underlying top-10 (so user can verify)
```

### Provider abstraction

```ts
// packages/core/src/rca/llm/provider.ts
export interface LlmProvider {
  name: string;
  /** Returns {prompt-tokens, output-tokens, content}. */
  call(opts: {
    system: string;
    user: string;
    model: string;
    maxTokens: number;
    /** JSON schema or a "respond as JSON only" hint. */
    responseShape?: "json" | "text";
  }): Promise<{ inputTokens: number; outputTokens: number; content: string }>;
}

// packages/core/src/rca/llm/anthropic.ts
export const anthropicProvider: LlmProvider = { ... };

// packages/core/src/rca/llm/openai.ts
export const openaiProvider: LlmProvider = { ... };
```

Auth via env: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`,
`OPENAI_BASE_URL` (for compatible endpoints).

No SDK dependency in core — both providers ship as ~100 LOC fetch wrappers.
This keeps the dep graph clean and matches how we ship eval-harness.

### Prompt structure

```
SYSTEM:
You are an RCA assistant. You will receive (a) a failure description, (b)
a ranked list of candidate locations from a static graph analyzer. Pick
the single most likely root cause. Be honest if no candidate is plausible
— output rootCause: null and explain.

USER:
## Failure
<failure description verbatim>

## Candidates (ranked by static analyzer)

### Candidate 1 (rank=1, score=0.7)
File: core/config.py:170
Symbol: __post_init__
Body:
```python
<30 lines>
```
Callers: [list]
Callees: [list]
Why this candidate ranked here: <causal scorer's reasoning>

### Candidate 2 …

## Output
Respond as JSON matching this schema:
{
  "rootCause": {
    "file": "<path relative to repo root>",
    "line": <int>,
    "symbol": "<name>",
    "hypothesis": "<≤3 sentences>",
    "confidence": <0..1>
  } | null,
  "alternatives": [{file, line, symbol, why}],
  "reasoning": "<1-2 sentence explanation of which candidate(s) you considered>"
}
```

### Token budget

- Failure description: untruncated (typically <500 tokens)
- 10 candidates × ~400 tokens (30 lines + metadata) = 4000 tokens
- System + schema + instructions: ~500 tokens
- Caller/callee list (truncated): ~1500 tokens
- **Total target**: ~6500 input tokens, ~1500 output → fits in 10K cap

If candidates push past budget: drop callers/callees first, then trim
body to 15 lines. Never truncate the failure description.

Per-call cost estimates (2026-05 pricing, Anthropic):
- Sonnet 4.6: ~$0.02 input + ~$0.02 output = ~$0.04/call
- Haiku 4.5: ~$0.005 input + ~$0.008 output = ~$0.013/call

For the eval (8 bugs × 4 modes including new `llm` mode = 32 calls),
default to Haiku to keep the eval bill under $0.50.

### Output schema (mandatory)

```ts
interface RcaLlmResult {
  // Inherited from non-LLM RcaResult — the ranked candidates the LLM saw.
  underlying: RcaResult;
  // What the LLM picked. Null = LLM said no candidate is plausible.
  llmVerdict: {
    rootCause: { file: string; line: number; symbol: string;
                 hypothesis: string; confidence: number } | null;
    alternatives: Array<{ file: string; line: number; symbol: string;
                          why: string }>;
    reasoning: string;
  };
  cost: { inputTokens: number; outputTokens: number; usd: number };
  provider: string;
  model: string;
  latencyMs: number;
}
```

Always return `underlying` even when LLM verdict is null — user can fall
back to the static ranking. This is part of the "honest tool" promise:
the LLM's pick is one signal, not the only signal.

## Eval plan

### Same corpus as Phase 1
`tools/eval/corpus-eval*.jsonl` — 8 bugs. Add `llm` mode to the harness:

```
node tools/eval/run-eval.mjs \
  --corpus tools/eval/corpus-eval-fixed.jsonl \
  --repo /path/to/repo \
  --modes current,text,file,llm,llm-cursor-baseline \
  --top-n 10
```

### New baseline mode: `llm-cursor-baseline`

This is the kill-criterion comparator. Implementation:
1. Send the failure description to the LLM
2. Provide it the entire repo's file tree (paths only, no bodies)
3. Let it ask for files via a tool-use loop OR provide top-K files by
   embedding similarity (whatever Cursor's `@codebase` does)
4. Get its verdict

Approximating `@codebase` honestly is hard. Simplest fair version: send
the failure + the file tree + 5-10 most-likely files (by some embedding
or BM25 retrieval over file names + first-50-lines). Let the LLM pick.

If the implementation effort for this baseline exceeds 1 day, we instead
**measure against the actual Cursor `@codebase` output by hand on a
sample of 3 bugs**. Less rigorous but cheaper and represents what users
would actually compare against.

### Acceptance metrics
- `text` (Phase 1): top-1 = 0.375 (the bar to beat)
- `llm` (Phase 2): must hit ≥0.5 top-1 to "show signal"
- `llm` vs `llm-cursor-baseline`: must be ≥10pp top-1 better — the kill
  criterion

If `llm` < 0.5 top-1: graph retrieval may not be feeding the LLM the
right candidates. Re-tune Phase 1's scorer rather than blame the LLM.

If `llm` ≥ 0.5 but < `llm-cursor-baseline + 0.1`: kill the bet. Either
re-architect or accept "cgrca is a calibrated KG" framing and stop.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LLM picks wrong candidate even when right one is in top-3 | Medium | High | Phase 1 still returns underlying ranking; users can verify |
| Token bill exceeds estimates on real repos | Medium | Medium | --max-llm-tokens cap is hard; truncate before exceeding |
| API auth/network flakiness | Medium | Low | --no-llm-fallback opt-in; default behavior degrades silently to Phase 1 |
| Cursor `@codebase` baseline approximation unfair | High | Medium | Hand-measure on 3 bugs as honesty floor |
| LLM hallucinates file/line not in candidate set | Low | Medium | Validate output against candidates; reject + retry once |

## What we DON'T do in Phase 2

- No streaming output (CLI is one-shot)
- No conversation history (each call is stateless)
- No fine-tuning (out of scope — Phase 5 self-improvement covers shots)
- No support for multiple LLM passes per bug (cost discipline)
- No caching of LLM responses (nice-to-have for re-runs but not Phase 2)
- No automatic retry on rate limit (just exit non-zero)

## Open questions to resolve before coding

1. **Provider default**: Anthropic Sonnet 4.6 vs Haiku 4.5? Recommend
   Sonnet for production CLI quality, Haiku for the eval to keep costs
   sane. Two defaults: `--model` per-call.
2. **Where does the API key live?** Just env var, or also a
   `~/.cgrca/config.json`? Env-only for v1.
3. **Failure on no API key**: error vs silent fallback to Phase 1?
   Default = error with clear instructions; `--no-llm-fallback` becomes
   moot. Re-evaluate after first user pain.
4. **MCP integration timing**: Phase 3 is designed separately. Phase 2
   ships ONLY the CLI; MCP gets the same pipeline in Phase 3.

## Implementation sequencing

1. Provider abstraction (`llm/provider.ts`, `llm/anthropic.ts`,
   `llm/openai.ts`) — 1 day
2. Prompt builder (`llm/prompt.ts`) reusing the existing `bodyPreview`
   and `callersOf`/`calleesOf` queries — 1 day
3. CLI flag wiring + JSON output shape — 0.5 day
4. Token-budget enforcer with degradation rules — 0.5 day
5. New eval harness mode `llm` (cost & latency tracking) — 1 day
6. Baseline mode `llm-cursor-baseline` (or hand-measure on 3 bugs) — 1-2 days
7. Run eval, write up results, decide gate — 0.5 day

**Total**: ~6-7 days agent time, fits the 1-2 week budget in the plan.
