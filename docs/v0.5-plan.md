# v0.5 plan — LLM-augmented RCA over the Halo KG

## Goal

Turn cgrca from a graph-walker that helps developers narrow already-localized
failures into a tool that takes a fuzzy bug description (prose, intent, partial
trace) and returns ranked suspect locations + a hypothesis — by feeding the
KG-narrowed sub-graph to an LLM. Pattern is GraphRAG specialized for code-RCA.

**Origin**: Phase-0 cold-start test (cyclist-marathon bug in athlai, 2026-05-02)
showed cgrca's existing input contract returns 0 candidates on prose-style
failure descriptions and 0 of 7 plausible cold-start anchors surfaced the actual
bug location. The KG was solid; the front door was narrow.

## Honest framing (until evaluation says otherwise)

- "RCA infrastructure for AI-built code, with optional LLM-augmented mode
  (experimental)"
- NOT "calibrated GraphRAG with a moat" — pre-Phase-4, that's marketing not
  evidence
- The KG itself is well-built. The retrieval-as-input-to-LLM bet has yet to be
  evaluated against alternatives (Cursor `@codebase`, raw Claude with whole
  repo, etc.)

---

## Phase 1 — Free-text retrieval (the unblocker)

**Effort**: 2-3 days
**Risk**: Low

### Ships
- Default mode for `cgrca rca`: when input doesn't match
  `symbol:`/`file:`/`test:`/stack-trace shapes, treat as free-text
- Tokenize: split on non-identifier chars, keep snake_case + camelCase +
  literal-quoted strings, drop English stopwords
- Cross-reference tokens against:
  - `symbols.name LIKE %token%`
  - `bodyPreview LIKE %token% AND LIKE %"literal"%`
  - `imports.local_name LIKE %token%`
- Score candidates by token-coverage + existing 7 signals
- **Fix the broken `file:` query** (currently returns null anchor) → returns
  all symbols in file ranked by 7 signals

### Eval gate (before Phase 2)
- Re-run cyclist-marathon + 5 other real bugs from athlai PR history
- Measure top-1 vs. baseline (current cgrca + naïve grep)

### Kill criterion
If text mode's top-1 isn't ≥2× current cgrca on this set → KG retrieval isn't
the bottleneck, something deeper is wrong. Stop, reassess.

---

## Phase 2 — LLM-augmented RCA

**Effort**: 1-2 weeks
**Risk**: Medium

### Ships
- New CLI: `cgrca rca-llm <failure>` (or `--llm` flag on `rca`)
- Pipeline:
  1. Run existing RCA → top-N candidates (N=10)
  2. Fetch each candidate's body (~30 lines via `bodyPreview`) + 2 hops of
     callers/callees
  3. Build structured prompt with failure + candidate set
  4. Send to LLM (Anthropic API + OpenAI-compatible — provider abstraction, no
     vendor lock)
  5. Parse response → (file, line, hypothesis, confidence)
  6. Return: LLM verdict + underlying candidate set so the user can verify
- Token budget: pre-trim each candidate to ~30 lines, total cap ~10k tokens
  (~$0.05-0.15 per call at current rates)

### Kill criterion
If `cgrca rca-llm` doesn't beat **"paste failure description + whole repo to
Claude with `@codebase`-style retrieval"** by ≥10pp top-1 → graph-walk
retrieval doesn't add value over embedding retrieval. Cursor wins this race.
Pivot or stop.

---

## Phase 3 — MCP integration

**Effort**: 3-5 days
**Risk**: Low

### Ships
- New MCP tool: `cgrca_rcaWithReasoning` — wraps Phase 2
- When called from inside Claude Code, uses the host's LLM (no API key needed,
  no extra cost to user)
- Outside Claude Code: env-configured Anthropic/OpenAI key

### Kill criterion
If users in the wild prefer calling Claude Code's native `@codebase`-style
retrieval directly → MCP wrapper is friction without value. Mark as
ship-but-deemphasize.

---

## Phase 4 — Honest evaluation

**Effort**: 1 week
**Risk**: This is the gate that decides everything

### Ships
- Test corpus: 20-30 real bugs from athlai + cgrca PR history (DIFFERENT from
  the 101-entry calibration corpus, to avoid overfit)
- Each entry: cold-start failure description + the merged-fix file:line
- Bench: cgrca-current vs. cgrca-text-mode (P1) vs. cgrca-llm (P2) vs.
  raw-Claude-with-whole-repo (baseline)
- Publish raw numbers in `CHANGELOG.md`. No selective reporting.

### Kill criterion (the hard one)
If final top-1 isn't ≥40% on real cold-start bugs → the whole bet (LLM-augmented
graph-walk RCA beats embedding retrieval) is wrong. Either re-architect or step
back to "cgrca is a calibrated KG you can use as input to your favorite tool"
and stop pretending it's an RCA product.

---

## Phase 5 — Self-improvement loop

**Effort**: 1 week
**Risk**: Low — but value depends entirely on Phases 1-4 working
**Conditional**: Only ship if Phase 4 shows real signal worth growing

### What it does
Closes the loop: every PR cgrca's bot comments on eventually merges with the
real fix. That's a labeled (failure → fix-location) tuple. Capture it, grow
the per-repo corpus, sharpen calibration over time.

### Ships
- GitHub-App logs every (failure, top-N candidates posted, merged-fix files)
  tuple to a per-repo `~/.cgrca/repos/<sha>/observations.jsonl`
- New CLI: `cgrca calibrate --from-observations` — ingests the observations,
  computes hit-rates per signal, optionally re-fits weights (with a "this run
  shifted weights by X% — apply?" gate so it's not silently changing scoring)
- For LLM mode: `cgrca calibrate --build-shots` extracts the highest-confidence
  past wins as 3-5 few-shot examples per repo, attached to future LLM prompts
- New CLI: `cgrca observations --misses` — bugs the bot consistently failed to
  localize. That's the next-architecture-iteration backlog, generated
  empirically.

### Where the real value lives
The **per-repo LLM few-shot library** is the part competitors can't easily
replicate. Cursor/Aider/Continue retrieve by embedding similarity each call —
they don't keep a structured observation log tied to outcomes. cgrca, with its
sqlite per repo + the github-app's PR-comment audit trail, has the substrate.

### Kill criterion
If observations after 3 months don't shift top-1 by ≥3pp on a rolling holdout
→ the loop isn't tightening fast enough. Either accept it as a "data archive"
(still useful for postmortems) and stop the auto-recalibration, or invest in
better label extraction (LLM-grade each observation: "is this PR really a bug
fix? Which file is THE fix?").

### Honest caveats
- Self-improvement is a multiplier, not a magic ingredient. ~15-20% better
  over a year. Does NOT rescue a structurally-wrong system.
- Signal is noisy. A PR labeled "fix bug" often touches 5 files, only 1 is THE
  fix. Easily 30% noise rate in labels without LLM-grading.
- Data velocity is slow. athlai gets ~10 fix-shaped PRs/week → ~500/year.
  Calibration shifts won't be dramatic.

## Risk register (honest)

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Cursor's `@codebase` is just as good or better | High | High | Phase 4 eval pre-commits to comparison. Lose openly if we lose. |
| Token bill is unfavorable for users | Medium | Medium | Default to running inside Claude Code (host's LLM, no extra cost). API path opt-in. |
| Graph retrieval ≠ better than embedding retrieval | Medium | High | Phase 1 eval catches before Phase 2 build |
| LLM hallucinates "the bug is in X" | Medium | Low | Structured output forces (file, line) commit; user verifies in seconds |
| Calibration corpus too small to claim "calibrated priors" | High | Medium | Honest disclosure in Phase 4. Either grow or drop the calibration claim. |
| Phase 5 self-improvement signal is too noisy | Medium | Medium | LLM-grade labels (`cgrca calibrate --grade-with-llm`) as a follow-up if needed |

---

## What we explicitly DON'T promise

- No "moat" framing in marketing until Phase 4 results justify it
- No Phase 2 build if Phase 1 eval fails — come back, ask, reassess
- No positioning shift in README until Phase 4 numbers are public
- No version-bump of "v0.5.0 GA" until at least Phase 1+2+3 ship and Phase 4
  passes eval. Pre-eval ships as `v0.5.0-rc.N` or `experimental` flag.
